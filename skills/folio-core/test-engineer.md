# Test Engineer Skill

## Role

Write and maintain unit tests for the content-authoring pipeline.
Tests are TypeScript using Bun's built-in test runner (`bun:test`).

## Test infrastructure

| Path | Purpose |
|------|---------|
| `scripts/tests/*.test.ts` | Test files (bun discovers `*.test.ts`) |
| `scripts/tests/helpers.ts` | Shared helpers: paths, project discovery, parsing |
| `scripts/tests/report.ts` | Test report generator (outputs `TestReport` JSON) |
| `scripts/tests/run-tests.sh` | Shell entry point (delegates to `bun test`) |
| `schemas/test-types.ts` | TypeScript types for test definitions, results, reports, coverage |
| `content/schema/` | Content-object types, Zod schemas, builder functions |
| `content/pipeline/validate.ts` | Content validation pipeline (schema + constraints + AST) |
| `content/pipeline/render.ts` | Block → output rendering + AST validation |
| `content/pipeline/build.ts` | Full document build from content objects |

## Dependency graph for scoped testing

Use the content `uses[]` fields (and any formalization-import graph)
to scope test runs:

1. **Identify changed blocks** — which `.ts`/`.md` files were modified
2. **Trace downstream dependents** — via `uses[]` in content objects
3. **Run targeted tests** — only validate/build the changed block + its dependents
4. **Report impact** — show how many blocks are affected by a change

The dependency graph comes from:
- `uses[]` field in content `.ts` manifests (narrative dependency)
- any formalization `import` statements (formal-layer dependency)
- `dependency-graph.svg` generated by CI

## When to write a test

1. **A bug was found** — write a regression test before fixing.
2. **A new script or config file is added** — test its contract.
3. **A rendering macro is changed** — test extraction still works.
4. **A new formal-layer project is added** — auto-discovered, but verify.
5. **A review skill detects a potential error** — it will ask the user
   whether to write a test. If yes, this skill writes it.

## How to write a test

```typescript
// In scripts/tests/<suite>.test.ts
import { describe, test, expect } from "bun:test";
import { REPO_ROOT, ... } from "./helpers";

describe("Feature under test", () => {
  test("specific behavior", () => {
    // Arrange
    // Act
    // Assert
    expect(result).toBe(expected);
  });
});
```

### Conventions

- **One test file per concern**: create new files for new concerns.
- **Use helpers.ts**: Don't hardcode paths or re-implement discovery.
- **Informational vs. blocking**: Tests that report data quality
  (coverage gaps, label mismatches) should log and pass. Tests that verify
  contracts (JSON validity, required fields, file existence) should fail.
- **Comment the failure point**: Each test should document what bug or
  failure it prevents.
- **Types from `schemas/test-types.ts`**: Use `TestResult`, `TestReport`,
  `CoverageEntry` etc. when generating structured output.

### Adding a new formal-layer project

When a new formal-layer library is added to the build manifest:
- `discoverProjects()` in `helpers.ts` auto-discovers it
- the project test suite automatically tests it (directory, files, audits)
- No manual test registration needed

### Test categories (from `TestCategory` type)

| Category | What it tests |
|----------|---------------|
| `formal-compile` | formal-layer files exist, have imports, are non-empty |
| `formal-library` | toolchain, build manifest, dependencies declared |
| `formal-gap` | unproved-gap count per project/file |
| `render-coverage` | render tags resolve, statement coverage, consistency |
| `doc-structure` | labels, cross-refs, environments |
| `schema-validity` | JSON manifests match their schemas |
| `infrastructure` | server config, scripts, deploy, .gitignore |
| `content-validation` | content objects pass Zod + constraint + AST checks |

### Content-object tests

Test content objects using the validation pipeline:

```typescript
import { validateObjects } from "../../content/pipeline/validate";

describe("Content objects", () => {
  test("all objects pass schema validation", async () => {
    const result = await validateObjects("content/<paper>/<section>/");
    const errors = result.issues.filter(i => i.level === "error");
    expect(errors).toHaveLength(0);
  });

  test("rendered output passes AST validation", async () => {
    // The validation pipeline includes AST checks automatically
    const result = await validateObjects("content/<paper>/<section>/");
    expect(result.valid).toBe(true);
  });
});
```

## Running tests

```bash
# All tests
bun test                              # from scripts/tests/
./scripts/tests/run-tests.sh          # from repo root

# By suite
./scripts/tests/run-tests.sh formal     # formal-layer project tests
./scripts/tests/run-tests.sh coverage   # render coverage
./scripts/tests/run-tests.sh infra      # Infrastructure

# JSON report (TestReport schema)
./scripts/tests/run-tests.sh --json
bun run report.ts --out test-report.json
```

## Report output

`bun run report.ts` outputs JSON conforming to `TestReport` in
`schemas/test-types.ts`. This report is consumed by:

1. **CI pipeline** — build status, pass/fail gates
2. **Publication pipeline** — coverage statistics for output artifacts
3. **Status overlay** — formalization/coverage status overlay

The report includes `by_category` breakdown so each pipeline stage
can extract just the tests relevant to it.
